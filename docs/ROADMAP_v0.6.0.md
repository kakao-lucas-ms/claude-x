# v0.6.0 Roadmap: Prompt Enhancement with External Best Practices

## 개요

사용자가 원할 때만 프롬프트를 "고도화"하여 실행하는 기능.
외부 베스트 프랙티스 DB를 활용해 Claude가 모르는 지식을 주입.

## 핵심 가치

| 기존 문제 | v0.6.0 해결 |
|----------|------------|
| 실시간 자동 개선은 흐름을 끊음 | Opt-in 방식으로 원할 때만 사용 |
| Claude가 이미 아는 내용 중복 | 외부 DB에서 새로운 지식 주입 |
| 사용자가 좋은 프롬프트를 모름 | 고도화된 프롬프트 보면서 학습 |

## 사용 시나리오

### Before (현재)
```
사용자: "이 코드 리뷰 해줘"
Claude: [기본적인 코드 리뷰 수행]
```

### After (v0.6.0)
```
사용자: "이 코드 리뷰 해줘" > 고도화해서 실행해줘

[claude-x가 베스트 프랙티스 DB 검색]
[유사도 높은 "코드 리뷰" 템플릿 발견]

고도화된 프롬프트:
"이 코드를 리뷰해줘:
 - 보안 취약점 (OWASP Top 10)
 - 성능 병목점
 - 에러 핸들링 누락
 - 테스트 커버리지
 - 코드 스타일 및 가독성
 - 잠재적 버그"

Claude: [포괄적인 코드 리뷰 수행]
사용자: (고도화된 프롬프트 보면서 학습)
```

## 트리거 방식

### 옵션 1: 자연어 (추천)
```
"코드 리뷰 해줘, 고도화해서"
"이거 수정해줘 > 베스트 프랙티스로"
"테스트 작성해줘 - enhance"
```

### 옵션 2: 명시적 명령
```
/enhance "코드 리뷰 해줘"
++코드 리뷰 해줘
@boost 코드 리뷰 해줘
```

### 옵션 3: MCP 도구
```
Claude가 analyze_and_improve_prompt 호출 후
사용자에게 "고도화된 버전으로 실행할까요?" 질문
```

### 템플릿 예시: code_review/general.md
```yaml
name: "Comprehensive Code Review"
intent: ["review", "코드 리뷰", "code review"]
template: |
  다음 관점에서 이 코드를 리뷰해줘:

  ## 기능성
  - 요구사항을 충족하는가?
  - 엣지 케이스 처리가 되어 있는가?

  ## 보안
  - 입력 검증이 적절한가?
  - 민감 정보 노출 위험은 없는가?

  ## 성능
  - 불필요한 연산이 있는가?
  - 메모리 누수 가능성은 없는가?

  ## 유지보수성
  - 코드가 읽기 쉬운가?
  - 적절한 추상화가 되어 있는가?

  ## 테스트
  - 테스트 커버리지는 충분한가?
  - 테스트하기 어려운 구조는 없는가?

tags: ["review", "quality", "security", "performance"]
quality_score: 9.2
usage_count: 1547
```

- [ ] CLAUDE.md에서 컨벤션 자동 추출

## 기대 효과

1. **사용자 학습**: 고도화된 프롬프트 보면서 자연스럽게 프롬프트 엔지니어링 학습
2. **품질 향상**: 베스트 프랙티스 기반으로 일관된 고품질 결과
3. **시간 절약**: 매번 상세 프롬프트 작성할 필요 없음
4. **Opt-in**: 원할 때만 사용하므로 흐름 방해 없음

## 성공 지표

| 지표 | 목표 |
|------|------|
| 고도화 사용률 | 프롬프트의 20% 이상에서 사용 |
| 사용자 만족도 | "유용했어요" 80% 이상 |
| 학습 효과 | 고도화 없이도 프롬프트 품질 향상 |
